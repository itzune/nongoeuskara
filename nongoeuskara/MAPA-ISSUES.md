# Nongoeuskara — Map-related issues & limitations

> **RESOLVED (2026-06-11).** All issues below were fixed at the root by
> replacing the artistic Wiki Commons SVG with a purpose-built map generated
> from real municipality boundaries (option **B/C** from §11). See
> "Resolution" at the end of this document.

Documenting the problems we've encountered trying to deliver a good map
experience for the "nongoeuskara" page. Almost all of them stem from the fact
that **the SVG we're using was never designed for programmatic interactivity**
— it's a static illustration authored in Inkscape circa 2010, uploaded to
Wiki Commons as a dialect zone map.

---

## 1. The SVG is a single-line file (1.2 MB)

The `map.svg` and `map_original.svg` files are each a single 1.2 MB line with
no line breaks. This is readable by browsers and Python's XML parser, but it
makes manual inspection / debugging extremely painful. Every `head`, `grep`,
and `xxd` hit shows truncated garbage.

**Impact:** Hard to debug structure. Had to write Python one-liners constantly
to explore the XML tree.

**Remedy (done):** Wrote `regenerate_map.py` which parses, enhances, and
rewrites the SVG. But the rewrite also produces a single-line file.

**Remedy (not done):** Could add `xml_declaration` formatting with indentation,
but the size stays the same and prettifying adds risk of breaking Inkscape
namespace handling.

---

## 2. SVG contains 4 different coordinate systems via `transform` matrices

The 1,579 paths use **10 different transform matrices** to scale/translate
coordinates. This means:

- `matrix(0.6861119, 0, 0, 0.686111, -45.93799, 63.534869)` — 314 paths (Zuberoa, Erronkari, etc.)
- `matrix(0.7340605, 0, 0, 0.7340597, 239.19777, 90.321086)` — ~232 paths (most Nafarroa)
- `matrix(1.0709746, 0, 0, 1.0709746, -127.1979, 186.99569)` — ~144 paths (Mendebalde/Bizkaia)
- 7 other unique transforms for small groups of paths (Lapurdi, isolated paths)

**Impact:**
- We cannot geocode towns to SVG coordinates with a simple linear
  projection. Each town needs the correct transform matrix applied.
- The `town_labels.json` file (375 towns with `{x, y}` coordinates)
  was generated by projecting Wikidata lat/lon through a single
  approximate transform, so **hover zones are positioned imprecisely**
  (especially in Bizkaia and Gipuzkoa).

**Remedy (tried):** The `town_path_map.json` maps 226 town names to their
SVG path IDs, but only works for towns whose paths have meaningful IDs.

## 3. Only 287 out of 1,579 paths have named IDs

The SVG contains:
- **287 named paths** — IDs like `Abaurregaina`, `Donostia`, `Lapurdi` (town names)
- **1,292 unnamed paths** — IDs like `path340`, `path15034` (generic polygons)

**Important:** The **named towns are almost entirely in Nafarroa**
(Healdeko/Iparraldeko Nafarroa Garaia + Burunda + Zaraitzu/Erronkari/Aezkoa).
Mendebalde (Bizkaia+Gipuzkoa+Araba): 435 paths, **all** unnamed.
Lapurdi: 24 paths, mostly unnamed.
Zuberoa: 36 paths, all unnamed.

**Impact:**
- Tooltip hover detection only works for 287 towns (Nafarroa-heavy).
- We added 375 invisible `<rect>` hover zones from Wikidata geocoding,
  but they're positioned imprecisely and overlap in dense areas.

---

## 4. Path rendering order hides smaller named towns

The SVG rendering order is layer-first, meaning:
1. Big regional polygon paths are drawn first (from the top of the file)
2. Smaller municipal paths are drawn later (within their `inkscape:label` groups)
3. But within each layer, *all* paths (including big ones) are drawn in
   document order

Result: When we `querySelectorAll("path")` and apply grey-out fills, the
larger polygon for the region often **covers** the smaller named town
polygon underneath. The town polygon *is* there but hidden by the bigger
one on top.

Additionally, many named paths are actually **overlapping sibling pairs** —
e.g., `path12438` (fill/stroke) and `path12440` (stroke-only outline)
for the same town. Both need to be styled together.

**Impact:**
- `elementsFromPoint()` helped somewhat (commit `942f5cb`) by looking
  through all elements under the cursor, not just the topmost.
- But the fundamental rendering issue remains: if a big layer polygon
  is highlighted, the smaller town-polygon beneath it is invisible.

## 5. The `data-model-label` mapping is coarse

We mapped 12 model labels to 12 layer groups, plus we flagged 73 individual
paths in Nafarroa with more specific labels. This gives us:

| Layer | Paths | Model label | Granularity |
|---|---|---|---|
| Mendebalde | 435 | `mendebal-sartaldea` | Single label for all Bizkaia/Gipuzkoa/Araba |
| Erdialde | 220 | `erdialde-sartaldea` (+ 10 paths with `erdialde-sortaldea`) | Mostly single label |
| Hegoaldeko Nafarroa Garaia | 375 | `nafar-erdigunea` (+ 5 specific paths) | Mostly single label |
| Zuberoa | 36 | `zuberera` | Correct but no per-town granularity |
| Lapurdi | 24 | `naflap-sartaldea` | Only 3 named paths |

**Key problem:** When the model predicts `mendebal-sortaldea` (eastern
Bizkaia/Gipuzkoa), we can't highlight only that sub-region — we have to
highlight the entire Mendebalde layer (435 paths).

Similarly, `erdialde-sortaldea` only maps to 10 specific named paths; the
remaining 210 Erdialde paths all default to `erdialde-sartaldea`.

**Impact:** The map experience can't show the fine-grained predictions the
model makes. The highlight is always layer-level, not sub-region level.

## 6. The SVG uses `clip-path` references

All paths use `clip-path="url(#SVG_CP_1)"` or `clip-path="url(#SVG_CP_1-1)"`.
These are defined in the `<defs>` section and clip path rendering to the
Euskal Herria boundary.

**Impact:** Not a bug per se, but if we ever try to use `innerHTML` to
insert the SVG (which we do in commit `2696707`), the clip-path references
*should* resolve correctly if the `<defs>` are present. However, if the
`<defs>` get stripped or ID conflicts arise (e.g., multiple SVGs on page),
clipping breaks and the map overflows its bounds.

**Current state:** We use `innerHTML` and it seems to work, but this is
fragile.

## 7. The original SVG had colorful `#ff8080` / `#ccaaff` fills

The original `map_original.svg` colored each dialect zone differently.
Our `regenerate_map.py` script remapped `data-model-label` attributes but
did **not** change the original `style="fill:#ff8080"` attributes. Instead,
our JavaScript `greyOutAll()` overrides them client-side with `#d0d5da`.

**Impact:**
- Browser loads the SVG with original colors, then JS paints over them.
  There's a brief flash of colored map before JS runs (FOUC).
- If JS fails to load or is slow, user sees the original colored map.
- The `map.svg` itself is 1.2 MB because it still carries all the
  original styling inline.

**Remedy (not done):** We could strip `style` attributes from `map.svg`
in the regeneration step and keep only `data-model-label`s, making the
file smaller and eliminating the FOUC.

## 8. Town geocoding: 375 towns with approximate SVG positions

We used Wikidata SPARQL to get coordinates for 417 Ahotsak towns and
projected them to SVG coordinate space using the formula:

```
svg_x = (lon - MIN_LON) / (MAX_LON - MIN_LON) * SVG_WIDTH * SCALE
svg_y = (MAX_LAT - lat) / (MAX_LAT - MIN_LAT) * SVG_HEIGHT * SCALE
```

This produces 375 entries in `town_labels.json` and 1,127 entries in
`town_coords_full.json`.

**Issues:**
- The projection assumes a single linear mapping, but the SVG uses
  multiple transform matrices. The projection is therefore accurate
  only for one region and drifts for others.
- 202 towns have no named SVG path at all (Mendebalde, Lapurdi, Zuberoa).
  We add invisible `<rect>` hover zones, but these overlap and misalign
  in dense areas like Gipuzkoa's coastline.
- 226 towns were matched to SVG path IDs via `town_path_map.json`, but
  many of those paths are **buried under larger polygons** (see issue #4).

## 9. Performance: 1,579 path elements styled via JS

Every `greyOutAll()`, `highlightLabel()`, and `resetHighlight()` call
iterates over all 1,579 `<path>` elements and sets their inline `style.fill`,
`style.stroke`, etc. individually.

**Impact:**
- On slower devices, highlighting a layer causes a noticeable repaint delay.
- The `mousemove` event fires on every pixel movement and calls
  `elementsFromPoint()`, which is inherently expensive.

**Mitigations done:**
- `elementsFromPoint` is called on the SVG root, not per-path.
- Debouncing could help but isn't implemented for hover.

---

## 10. The SVG path data uses relative coordinates

All paths use `m` (relative moveto) and relative commands (`l`, `c`, etc.),
not absolute. This makes it extremely hard to:
- Programmatically modify individual path coordinates
- Animate specific town boundaries
- Compute bounding boxes for per-town hit testing

The transform matrices make this even harder — a path's `bounds` in
screen-space needs the transform applied first.

---

## 11. Summary: we need a proper SVG

The root cause of most issues is that we're using an **artistic/illustrative
SVG** as if it were a **geospatial data format**. What we actually need is
one of:

| Option | Effort | Quality |
|---|---|---|
| **A.** Hand-crafted SVG where every municipality is a named `<path>` with a single coordinate system and correct `data-model-label` | Very high (manual work for 375+ towns) | Excellent — fully interactive |
| **B.** GeoJSON from OSM/Wikidata boundaries, rendered via Leaflet/D3 | Medium-high | Excellent — zoomable, accurate |
| **C.** Simplify current SVG: strip original styling, merge paths by model-label into one `<path>` per zone, keep named-town `<path>`s in a separate overlay layer | Medium | Good — cleaner code, fewer elements |
| **D.** Status quo + incremental fixes | Low (what we're doing) | Mediocre — fragile, coarse, biased toward Nafarroa |

**The current approach (D)** works for an MVP but will never produce a
polished, accurate map experience. The biggest UX issues today:
- Hover tooltips don't work for Bizkaia/Gipuzkoa towns (only via imprecise rects)
- Layer-level highlighting can't show fine-grained model predictions
- Path overlap means some towns are permanently invisible/drowned out
- Geocoded hover zones misalign with actual map features

## 13. Resolution (2026-06-11): regenerated map.svg from real boundaries

Instead of patching the Inkscape illustration, `map.svg` is now **generated**
by `build/build_map.py` from geospatial sources:

- **Boundaries:** opendatasoft `georef-spain-municipio` (Araba, Bizkaia,
  Gipuzkoa, Nafarroa — 543 municipalities) + `france-geojson`
  (159 Iparralde communes). Download with `build/fetch_data.sh`.
- **Basque names:** Wikidata `eu` labels keyed by INE/INSEE code.
- **Dialect labels:** the 430 Ahotsak town→azpieuskalki assignments mapped to
  the 12 tier-3 model labels. **All 430 towns matched** (a handful of name
  collisions like Getaria (L)/Getaria (G), Jatsu Garazi/Jatsu, Lekunberri
  (NB)/Lekunberri (N), Bastida/Labastida are handled with per-code overrides).
- **Geometry:** topology-preserving simplification (`shapely.coverage_simplify`)
  so shared borders stay shared; equirectangular projection with latitude
  aspect correction.

The output is a 407 KB (112 KB gzip) SVG where **every municipality is one
named `<path>`** in a single coordinate system:

```svg
<path id="oiartzun" class="mun" data-name="Oiartzun"
      data-model-label="erdialde-sortaldea" d="M…"/>
```

How each old issue is resolved:

| # | Issue | Resolution |
|---|---|---|
| 1 | single-line 1.2 MB file | generated, one path per line, 407 KB |
| 2 | 10 transform matrices | single coordinate system, zero transforms |
| 3 | only 287/1579 named paths | all 702 paths named with `data-name` |
| 4 | overlapping paths hide towns | polygons tile the territory, no overlap |
| 5 | coarse label mapping | per-municipality labels for all 430 Ahotsak towns — `mendebal-sortaldea`, `erdialde-sortaldea`, etc. highlight exactly their towns |
| 6 | clip-path fragility | no clip-paths |
| 7 | FOUC from original fills | no inline styles; all styling via CSS |
| 8 | imprecise geocoded hover rects | removed; hover hits the real polygon |
| 9 | JS styles 1,579 paths per update | highlight = one `data-active-label` attribute on the SVG root + static CSS rules (O(1)) |
| 10 | relative path data + transforms | clean generated path data |

`src/map.js` was rewritten accordingly: CSS-driven highlighting, native
per-polygon hover (tooltip shows town + dialect zone), light/dark theme via
CSS variables. To regenerate the map:

```bash
cd nongoeuskara/build
./fetch_data.sh        # downloads boundary + Wikidata name data
python3 build_map.py   # writes ../map.svg + build_report.json
```
